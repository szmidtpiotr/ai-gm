GAP 1 — Roll cards nie wyświetlają się w combat (PRIORYTET)

Problem: actions.js parser roll cards szuka markera __AI_GM_ROLL_V1__
(narracyjny), ale combat wysyła __AI_GM_COMBAT_ROLL_V1__ (inny marker).

Verify:
  grep -n "AI_GM_ROLL_V1\|AI_GM_COMBAT_ROLL_V1" frontend/js/actions.js frontend/js/ui.js

Fix: w parserze roll cards w actions.js — obsłuż OBA markery:
  if line.includes("__AI_GM_ROLL_V1__") || line.includes("__AI_GM_COMBAT_ROLL_V1__")

Dane z __AI_GM_COMBAT_ROLL_V1__ JSON (już dostępne w user_text):
  d20, modifiers[], total, hit, damage, enemy_key, target_name,
  enemy_dead, combat_victory, summary_line, attack_label

Roll card dla player_attack powinna pokazać:
  [attack_label]  d20: [d20] + [mod] = [total]
  ✅ TRAFIENIE / ❌ PUDŁO  →  obrażenia: [damage]
  (jeśli enemy_dead: true → dopisz "💀 Bandyta pokonany")

Roll card dla enemy_attack (z enemy-turn response):
  [ATAK WROGA]  d20: X  →  ✅/❌  →  obrażenia: Y
  (użyj danych z response body POST /combat/enemy-turn)

DO NOT remove existing __AI_GM_ROLL_V1__ handling — keep both.

### GAP 1 — karty rzutów walki (`__AI_GM_COMBAT_ROLL_V1__`)
- W **`ui.js`**: `_extractNextCombatRollPayload`, `stripCombatRollBlocksForRollCue`, `buildInterleavedNarrativeAndCombatHtml` — osadzone bloki walki w narracji GM są składane z kartami + ostatni fragment tekstu przechodzi przez `parsePendingRoll` (cue „Roll … d20”).
- **`finalizeStreamingBubble`**: przy narracji z markerem walki zastępuje `<pre>` treścią z `buildInterleaved…`; przy `_combatJustEnded` nadal **nie** podnosi popupu rzutu (czyści `pendingRoll` / `activeRollRequest`).
- W **`actions.js`**: `parsePendingRoll` ignoruje fragmenty z `__AI_GM_ROLL_V1__` / `__AI_GM_COMBAT_ROLL_V1__` (przez `stripCombatRollBlocksForRollCue`), żeby wykrywać końcówkę „Roll …”.
- **`buildCombatRollCardHtml`**: `player_attack` — AC (`target_ac`), pudło/unik/nat1, stopka „Wróg pokonany”; poprawione liczenie AC (`acNum`).

### GAP 2 — łupy po victory
- Błąd logiki na **`[DONE]`**: wcześniej przy `_combatJustEnded` wywoływano `consumeCombatJustEndedGuard()` → **`cancelDeferredVictoryUi()`** i **nie** wywoływano `showVictoryAfterNarration` — to kasowało m.in. `_pendingLoot`. Teraz przy `_combatVictoryUiPending` zawsze jest **`await showVictoryAfterNarration(...)`** (oba miejsca: normalny `[DONE]` i stream bez `[DONE]`).
- W **`applyCombatEndedSseToken`**: jeśli payload ma `loot_pool`, trafia do `combatPanel._pendingLoot` (na wypadek rozszerzenia API).

### GAP 3 — tury w panelu
- **`combat_panel.js`**: sekcja **`#combat-engine-turns`**, `GET /combat/turns`, format wierszy **atak gracza / atak wroga / śmierć** zgodnie ze schematem z dokumentu.

### GAP 4 — panel nad kartą postaci
- **`index.html`**: **`#combat-panel-slot`** jest **nad** `#sheet-panel-body`.
- **`styles.css`**: przy otwartym arkuszu `.sheet-panel` używa **`display: flex; flex-direction: column`**, slot walki `flex: 0 0 auto`.
